SparkCore学习笔记

发表于 2020-05-23 更新于 2020-06-28 分类于 计算机 ， 分布式计算 ， Spark 阅读次数：

Spark是基于内存的计算框架，它主要包括SparkCore，SparkSQL，SparkML，SparkStreaming等方面的内容，本片文章从SparkCore的角度出发，从与MR的区别，运行模式，核心RDD，代码流程，资源和任务调度，Shuffle等方面来介绍Spark

Spark与MR的区别

  1. Spark基于内存迭代计算, MR是基于磁盘迭代计算
  2. Spark中有DAG（有向无环图）
  3. MR中只有map和reduce算子（相当于Spark中的map和reduceByKey），而Spark中有很多应用不同场景的算子

Spark运行模式

  1. Local：用于本地测试，在IDEA中经常使用Local模式
  2. Standalone：Spark自带的资源调度框架，支持分布式搭建
  3. Yarn：Hadoop生态圈中的资源调度框架，Spark可以基于Yarn运行
  4. Mesos：资源调度框架

Spark核心RDD

RDD（resilient distributed dataset）也叫弹性分布式数据集

RDD的创建

接下来通过scala的伪代码来说明RDD创建的3种方式

RDD的五大特性

  1. RDD是由partiton构成
  2. 算子（函数）作用在partition上
  3. RDD之间有依赖关系
  4. 分区器作用在（K，V）格式的RDD上
  5. partition对外提供最佳计算位置，利于数据处理的本地化

RDD注意事项

  1. textFile读取HDFS文件的方法的底层原理是什么？

    该方法的底层是通过调用MR来读取HDFS文件，即要进行split，每个split对应一个block，也对应一个partition

  2. 什么是（K，V）格式的RDD？

    RDD中的数据是一个个的二元组

  3. 哪里体现了RDD的弹性（容错性）？

    • RDD的分区可多可少
    • RDD之间有依赖关系

  4. 哪里体现了RDD的分布式？

    RDD的partition是分布在多个节点（机器）上的

持久化算子

cache

  1. cache算子默认将数据存在内存中

  2. cache() = persist(StorageLevel.MEMORY_ONLY)

persist

  1. 可以手动指定持久化级别
  2. 常用的持久化级别（存储类型）：
    • MEMORY_ONLY
    • MEMORY_ONLY_SER
    • MEMORY_AND_DISK(先存到内存，内存满了在溢写磁盘)
    • MEMORY_AND_DISK_SER
  3. 存储级别的后缀”_2” 表示有副本
  4. 尽量少使用DISK_ONLY的持久化级别

checkpoint

  1. 使用场景

    当lineage（RDD之间的依赖关系构成的链） 计算非常复杂，可以尝试使用checkpoint将数据直接持久化到指定的目录

  2. 执行流程

    step1：当spark job完成之后，Spark 会从后往前回溯，找到带有checkpoint的RDD并给它做标记
    step2：回溯完成后，Spark框架会重新启动一个job,计算标记的RDD的数据，放入指定的checkpoint目录中
    step3：数据计算完成并放入目录之后，会切断RDD的依赖关系，当Spark Application执行完成之后，数据目录中的数据不会被清除

注：checkpoint要指定存储目录！！！

持久化算子之间联系和区别

  1. cache,persist,checkpoint 都是懒执行，最小持久化单位是partition
  2. cache和persist之后可以直接赋值给一个值，下次直接使用这个值（即持久化的数据），但是这个时候后面不能紧跟action算子
  3. cache和persist的数据，当application执行完成之后会自动清除，但是通过checkpoint持久化到数据目录中的数据不会被清除

Stage

  1. RDD的宽窄依赖是什么？

    • 宽依赖（shuffle依赖）：父RDD与子RDD的partition之间的关系是一对多
    • 窄依赖：父RDD与子RDD的partition之间的关系是一对一，或多对一
      注：对应关系是从父RDD的角度来讨论的

  2. 相关术语有哪些？

    • 资源层面
      • Cluster Manager：在集群上获取资源的外部服务（比如Standalone, Yarn,Mesos）
      • Driver Program：用来连接工作进程（Worker）的程序
      • Master（Standalone）：资源管理的主节点（进程）
      • Worker（Standalone）：资源管理的从节点（进程）或者称为管理本机资源的进程
      • Executor：是在一个Worker进程所管理的节点上为某个Application启动的一个进程，该进程负责执行Task，并且负责将数据存在内存或磁盘上，每个应用都用各自独立的executor
    • 任务层面
      • Application：基于Spark的用户程序，包含了driver程序和运行在集群上的executor程序
      • Job：包含很多Task的并行计算，和Action算子一一对应
      • Stage：一个Job会被拆分成多组Task，每组Task就称为Stage（就像MapReduce分Map Task和Reduce Task）
      • Task：被送到Executor上的工作单元

  3. Stage是怎么产生的？
    Stage是有一组并行的Task组成，RDD内不存数据，partition也不存数据，partition中存的是逻辑（算子） 根据RDD之间的宽依赖可以划分成不同的stage

  4. Spark计算模式是什么？
    管道（pipeline）计算模式，即f3(f2(f1(textFile…))), 1+1+1 = 3的形式

  5. 管道数据合适落地？

    • shuffle write时
    • 对RDD进行持久化时

  6. Stage的并行度由什么决定？
    由finalRDD的partition个数决定

  7. 如何提高Stage的并行度？
    增大RDD的partition个数，常用的算子有：

    • reduceByKey(xxxx, numPartition)
    • distinct(xxxx,numPartition)
    • join(xxxx,numPartition)
    • repartition

      Spark任务调度

Spark资源调度和任务调度

资源调度流程

  1. 集群启动，Worker向Master汇报资源，Master掌握集群资源
  2. 当 new SparkContext 时，会启动Driver，并创建 DAGScheduler 和 TaskScheduler
  3. TaskScheduler 向 Master 申请资源，Master 找到满足资源的节点，启动 Executor
  4. Exeuctor启动之后反向注册给Driver，Driver掌握了一批计算资源

任务调度流程

  1. Action算子触发Job，Job中RDD之间的依赖关系形成DAG有向无环图（即lineage构成的图）
  2. DAGScheduler 拿到DAG后按照RDD之间的宽窄依赖关系切割每个Job，划分Stage，将Stage以TaskSet形式提交给TaskScheduler
  3. TaskScheduler遍历TaskSet，拿到一个个的Task，发送到Executor中的线程池中执行
  4. TaskScheduler监控Task，回收结果

调度流程图

粗力度细粒度资源申请

粗粒度申请

  • 当application执行之前首先将所有的资源申请完毕，如果申请不到一直处于等待状态，直到自愿申请到为止。必须一次申请完毕之后才会执行任务。这样，task执行的时候就不需要自己申请资源，加快了task的执行效率。task执行快了，Job就快了，进而application就快了。必须要等最后一个task执行完毕之后，资源才会被释放
  • 优点：Application执行快
  • 缺点：集群资源不能充分利用

细粒度申请

  • 当application执行之前不会将所有的资源申请完毕，Task执行的时候需要自己申请资源，自己释放资源，这样Task执行相对慢
  • 优点：集群资源可以充分利用
  • 缺点：application执行相对慢

SparkShuffle

Shuffle的提出

reduceByKey会将上一个RDD中同一个key对应的value聚合并计算生成新的value，进而生成一个新的RDD[(key,value)]。聚合之前，因为RDD是弹性分布式数据集，RDD的partition极有可能分布在各个节点上，所以每一个key对应的value不一定都是在一个partition中，也不太可能在同一个节点上

如何针对同一个key进行聚合呢？

  • Shuffle Write：上一个 stage 的每个 map task 就必须保证将自己处理的当前分区的数据相同的key写入一个分区文件中，可能会写入多个不同的分区文件中（因为可能有多个key）
  • Shuffle Read：reduce task 就会从上一个 stage 的所有 task 所在的机器上寻找属于自己的那些分区文件，这样就可以保证每一个 key 所对应的 value 都会汇聚到同一个节点上去处理和聚合

Shuffle 指的是节点之间数据传输的过程。常见的shuffle算子有 xxx byKey 的，distinct，还有各种 join 。Shuffle类算子可以自己指定分区数量，比如 join 后的默认分区数量是多的那个rdd的分区数量，也可以自己指定分区数量。

Spark中有两种shuffleManager，在Spark 1.2之后引入的SortShuffleManager，1.6默认使用的是SortShuffleManager，2.0+HashShuffleManager 被丢弃

HashShuffleManager

普通机制

  • 示意图

  • 流程

    1. map task处理完数据之后写往buffer缓存区，默认大小为32k，写往buffer缓冲区的个数与 reduce task 个数一致（PS：buffer起到数据缓存的作用）
    2. 缓冲区满 32k 溢写磁盘，每个buffer文件（缓存区）对应一个磁盘小文件
    3. reduce task 来拉取数据对应的磁盘小文件

  • 问题

    产生磁盘小文件过多，会导致以下问题：

    1. 在Shuffle Write过程中会产生很多写磁盘小文件的对象
    2. 在Shuffle Read过程中会产生很多读取磁盘小文件的对象
    3. 在 JVM 堆内存中对象过多会造成频繁的gc，gc还无法解决运行所需要的内存的话，就会OOM
    4. 节点之间拉取数据的连接多，在数据传输过程中会有频繁的网络通信，频繁的网络通信会导致出现通信故障的可能性大大增加，遇到网络连接不稳定导致拉取数据失败的概率大，会加大数据处理的时间。一旦网络通信出现了故障会导致shuffle file cannot find，由于这个错误导致的task失败，TaskScheduler不负责重试，由DAGScheduler负责重试Stage

  • 总结

    1. 产生磁盘小文件的个数是M（map task的个数）*R（reduce task的个数）
    2. map task的计算结果会根据分区器（默认是hashPartitioner）来决定写入到哪一个磁盘小文件中去。reduce task会去 map 端拉取相应的磁盘小文件。

合并机制

  • 示意图

  • 流程

    1. map task处理完数据之后写往buffer缓存区，默认大小为32k，写往buffer缓冲区的个数与 reduce task 个数一致
    2. 同一个 core 中的task 共用一份 buffer 缓存区
    3. 缓存区满32k溢写磁盘，每个buffer缓存区对应一个磁盘小文件
    4. reduce 端拉取数据

  • 问题

    相对于普通情况，shuffle文件大大减少，当reduce task 个数多，或者core的个数多的时候，产生磁盘小文件的个数还是比较大

  • 总结

    产生磁盘小文件的个数是C(core的个数)*R（reduce的个数）

SortShuffleManager

普通机制

  • 普通机制示意图

  • 流程

    1. map task 的计算结果会写入到一个内存数据结构里面，内存数据结构默认是5M
    2. 在shuffle的时候会有一个定时器，不定期的去估算这个内存结构的大小，当内存结构中的数据超过5M时，比如现在内存结构中的数据为5.01M，那么他会申请5.01*2-5=5.02M内存给内存数据结构
    3. 如果申请成功不会进行溢写，如果申请不成功，这时候会发生溢写磁盘
    4. 在溢写之前内存结构中的数据会进行排序分区
    5. 然后开始溢写磁盘，写磁盘是以batch的形式去写，一个batch是1w条数据
    6. map task执行完成后，会将这些磁盘小文件合并成一个大的磁盘文件，同时生成一个索引文件
    7. reduce task去map端拉取数据的时候，首先解析索引文件，然后根据索引文件拉取对应的数据

  • 总结

    产生磁盘小文件的个数是2*M（map task的个数）

广播变量

  1. 提出背景

    如果每一个task都要用到list的时候，那么当task的数量过多的时候就会造成资源浪费的问题，比如：100个task，就要传输100个list，并且都要保存在executor中，这样就会造成大量资源（内存）被占用。

  2. 广播的引入

    为了解决上述问题，引入了广播变量。将变量广播之后，在在Executor中的block manager中就有两相应的广播变量，之后每次task传输的时候不需要携带广播变量，如果在Executor中需要广播变量的话，那么就可以block manager中寻找需要的广播变量

  3. 示意图

  4. 语法格式

  1. 注意
    • 不能将RDD广播出去，但是可以将RDD的结果广播出去，比如：可以将rdd.collect()广播出去
    • 广播变量在Driver中定义，在Executor中使用，但是绝不能在Executor中改变广播变量的值
