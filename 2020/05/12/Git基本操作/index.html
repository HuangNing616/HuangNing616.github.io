Git基本操作

发表于 2020-05-12 更新于 2020-05-25 分类于 计算机

前言

Git 是一种版本控制系统（version system control，简称VSC)，VSC可以记录文件在一段时间内的变化。

文件概念图

为了便于理解git在本地上的操作，我们需要了解下方的文件概念图

其中.git history 等价于提交图(commit graph)， 它被放在隐藏文件.git中。.git history包含构造repo需要的metadata和数据库对象。 如果我们将.git 发送给用户A，那么用户A就相当于有了整个git项目。

Git 信息配置

  1. git confit —global user.name “globalhuangning” 配置全局用户姓名
  2. git config —global user.email “globalhuangning@123.com” 配置全局用户邮箱
  3. git config —local user.name “huangning” 配置局部用户姓名
  4. git config —local user.email “huangning@qq.com“ 配置局部用户姓名
  5. git config —list：查看配置列表

Git 本地常用操作

  1. git init：创建一个空的repository（简称repo）
  2. git status：查看当前分支以及working tree 和 staging area 中文件的状态
  3. git add xx：将working tree中的xx文件放到staging area中, 使其变成tracked/将被提交的文件
  4. git add . ：将working tree中的所有文件放到staging area中, 使其变成tracked/将被提交的文件
  5. git add s*：将working tree中首字母为s的文件放到staging area中, 使其变成tracked/将被提交的文件
  6. git rm —cached xx：将staging area中的xx文件再次变成unstaged/untracked文件
  7. git diff：查看workingTree 和 staging area 之间文件的区别
  8. git diff —staged：查看staging area和最近一次commit的文件的区别
  9. git commit -m “xxx”：将staging area中的文件提交到history中，并且提交信息为xxx
  10. git commit：同上，但是可以将提交信息编辑成多行的形式
  11. git log : 提交日志，其中最新的提交显示在最顶端，提交信息包括哈希码，用户名，邮箱以及相应的提交描述
  12. git log — xxx文件：显示和xxx文件相关的提交日志
  13. git log —all —decorate —oneline —graph: 可视化提交图

补充：

  1. commit 和 add 可以写在一行，中间用；隔开
  2. 如果针对所有文件进行add和commit，可以用 git commit -a -m “xxx”
  3. working tree（working directory）中不包括.git子目录

  4. 可以通过重命名的方式来方便绘制可视化：alias graph=”git log —all —decorate —oneline —graph”

  5. git rm -r —cached . 将所有文件都变成为untracked（在用.gitignore的时候可能会用到，因为有的时候不小心将本该忽略的文件加入了版本控制，这样就需要将加入版本控制的文件变成untracked）

Git branch的使用

A branch is a small label points to a commit. HEAD pointer tells us what we have checked out in Git，通常情况下 HEAD 会指向某个branch，它是一种symbolic pointer

  1. git branch <分支A>：创建分支A
  2. git branch：展示现在存在那些分支
  3. git checkout <分支名称>：切换到哪个分支上
  4. git commit -a -m “auth for s1” 将任何tracked并且被修改的文件add到stageArea并且被提交!!!
  5. git branch —merged：显示出哪些分支进行了合并（之后好删除分支）
  6. git branch -d <分支A>：删除分支A

  7. git branch -D <分支A>：可以删除还没有合并的分支A

  8. git merge 分支名成：合并

补充：

  • 两种merge方式：

    • fast-forward merge：被merge的分支包括master分支上的历史提交信息

    • 3-ways merge：被merge的分支和master分支在不同路径上，其中3-ways表示从同一个节点分到不同的节点再到新的提交节点上

  • 每次切换到一个分支，workingtree和staging area中的文件都会改变（update working tree 和 staging area）

解决基本的merge冲突：比如修改了相同行或者相同文件的时候？
创建并且切换到分支的快捷命令：git checkout -b dev
针对与base commit而言，某一行的change和no-change,git会将前者作为获胜者，也就是merge的时候会change
如果同一行两个分支有不同的change，那么git就不能猜到我们到底需要哪种改变
如果同一行两个分支有相同的change，那么就不会引起冲突
如果同一行其中一个分支有change，但是另外一个分支有delete，那么就不会引起冲突
git merge —abort：发现merge冲突之后，终止merge（前提是我们的working tree 和stage area都是干净的）

将HEAD直接指向某个提交(我们可以直接切换到某次提交而不是某个分支)
我们就有了一个 detached HEAD state
语法：git checkout 87abc55（hash码前七位）

最后一点：

  1. 如果干净的工作区和stage area不满足的话，我们就不能切换到其他分支或者很难merge

git stash: 通过存储当前信息，实现了切换分支或者merge等功能
git stash list (展示所有贮藏的简短信息)
git stash list -p (展示所有贮藏的完整信息)
重新应用这些stash
git stash apply(会应用最新的一次stash)
而且应用这些stash之后原来的stash list也不会pop或者remove，所以之后仍然可以重新使用这些stash
git stash pop: 删除stash列表
git stash apply 相应的label: 使用某个指定label的stash
git stash save 描述信息：为每个stash label加上描述性语言

============================git remote=============================================
remote is simply(只是) a repository in another location(可以是git服务器，可以是github上的fork，也可以是自己本机的其他repo)

  1. git clone 将会retrieve/copy github上的工程到本地(clone 需要工程的url，有两种：ssh和html)
  2. git fetch: 将会retrieve 或者 fetch update of repo on GitHub
  3. git push: 将会upload or push updates to the remote in github
    当我么fetch或者push的时候不需要记得full location fo our remote 因为git可以起一个简短的名字或者别名alias（别名）
  4. git remote: 返回远程连接的名字
    出现origin, 表示clone之后，origin是我们第一次远程连接的默认名称
    git remote -v: 显示远程连接的名称和完整地址

git pull = git fetch && git merge
git pull —rebase = git fetch && git rebase
git log —oneline —graph使得提交可视化

  3. git fetch origin (然后我们可以git merge origin/master 将本地的master也指向最新的commit）
  4. git pull == git fetch origin + git merge(建议不要这么做！！！还是分开写好，避免出错，这样又一个清晰的工作流程)
  5. 本地修改之后，本地的master会领先github上repo的master
  6. git push origin：将本地最新的提交（本地的改变）上传到github(push 依赖于github的authentication,这个需要事前设定好SSH 的authentication)

  1. How to undo a working tree change?（恢复working tree中的文件）
    git checkout — xxx文件
    stageArea中的xxx文件覆盖掉到workingTree中经过修改的xxx文件
    ps:注意，经过此操作之后，之前在workingTree中针对s1的糟糕操作就不可以还原了！！！）

  2. How to undo staging of file? (如何恢复（unstaged）stageArea中的文件?)
    git reset HEAD(表示最近一次提交) xxx文件
    表示将最近一次提交的xxx文件覆盖到StageArea(IndexArea)中的xxx文件，这样就做到了恢复stageArea中的xxx文件
    ps：注意，这样操作只恢复（restore）了stageArea中的xxx文件，对于workingTree中的xxx文件还是最新版本，所以还需要git checkout — xxx文件来恢复working tree中的文件

更多的撤销（undo）操作（恢复操作）

  1. restore from an early commit(恢复更早的提交中恢复文件）
    git checkout xxxxx(5位16位哈希码就可以了) — s2
    这就将xxx码对应的那次提交中的s2文件同时恢复到stage area 和 working tree中

Linus常用命令

  1. ls: 显示文件列表
  2. ls -a： 显示隐藏文件（以’.’点符号开头的文件名）
  3. ls -R: 递归显示所有子目录和子文件
  4. ls -al: 显示出文件的详细信息，详细信息会以列状格式显示出来，每列的含义如下所示:
  • 第一列：文件类型和访问权限
  • 第二列：文件的硬链接数
  • 第三列：文件的拥有者
  • 第四列：文件所属用户组
  • 第五列：文件的字节大小
  • 第六列：文件最后的修改时间
  • 第七列：文件名或目录名
  1. alias graph=”git log —all —decorate —oneline —graph” 给命令起一个外号
  2. cat 文件：查看文件
  3. more/less 文件：查看文件
